## 二、今天完成的主要工作

### 1. 项目认证能力继续完善

围绕登录、注册、验证码、邮箱发送这条链路，补齐并修正了认证服务层和接口层的实现。

当前认证相关接口已经形成以下结构：

- `GET /auth/captcha`
  - 获取账号密码登录使用的图形验证码
- `POST /auth/login`
  - 账号密码登录
  - 需要提交 `userName`、`password`、`captchaKey`、`captchaCode`
- `POST /auth/email/code`
  - 发送邮箱验证码
  - 支持 `LOGIN` 和 `REGISTER` 场景
- `POST /auth/login/email`
  - 邮箱验证码登录
- `POST /auth/register`
  - 用户注册

认证链路中已具备以下能力：

- 图形验证码生成并写入 Redis
- 图形验证码登录前校验，并在校验成功后删除，避免重复使用
- 邮箱验证码生成并写入 Redis
- 邮箱发送频率控制
- 注册时校验邮箱验证码
- 邮箱验证码登录时校验邮箱验证码
- 账号密码登录时通过 `PasswordEncoder` 做密码匹配
- 登录成功后签发 JWT Token

### 2. 邮箱验证码发送能力接入并优化

邮件能力已经接入项目，配置和调用链路都已打通。

今天重点做了两件事：

- 保留邮件验证码发送能力
- 将邮件发送改为异步，减少接口等待时间

具体调整如下：

- 在启动类中开启了 `@EnableAsync`
- 邮件发送服务 `MailService` 改为异步发送
- 邮箱验证码接口从“同步等待邮件发送完成再返回”改为“先写 Redis，再异步发送邮件”

这样改完后，接口响应速度会明显快于之前的同步 SMTP 模式。虽然 SMTP 服务器本身的发送速度仍然受外部网络和 163 邮箱服务影响，但用户请求线程不再被邮件投递过程阻塞。

### 3. Redis 验证码缓存规则继续完善

今天重点确认并收口了认证相关缓存 Key 的用途，主要用于以下数据：

- 登录图形验证码
- 邮箱登录验证码
- 邮箱注册验证码
- 邮箱发送频率限制
- 登录 Token 缓存

这部分能力主要依赖：

- `RedisCache`
- `CacheConstants`
- 认证服务中的验证码校验逻辑

这一步的意义是让后面继续扩展登录态、登出、验证码频控、邮箱重试等功能时，有稳定的缓存前缀规范可复用。

### 4. DTO / VO / Entity 分层继续落地

今天继续明确并验证了项目的对象分层：

- `Entity`
  - 数据库表映射对象
- `DTO`
  - 接口入参对象
  - 用于参数校验
- `VO`
  - 接口返回对象
  - 用于控制对前端暴露的数据范围

这一步非常关键，因为它纠正了之前“把接口校验和返回都堆在实体类上”的做法。当前项目已经明确后续开发规范应为：

- 不再把参数校验主要写在数据库实体类上
- 新增和修改接口优先单独建 DTO
- 返回前端的数据优先单独建 VO

这为后面继续补用户、角色、文章、分类、标签等业务提供了比较稳定的代码结构。

### 7. 编码问题排查与处理

今天还遇到了一个比较隐蔽的问题：修复乱码后，部分 Java 文件因为保存成了带 BOM 的 UTF-8，导致 Maven 编译失败。

具体表现为：

- 编译器报错 `非法字符: '\ufeff'`

最终已经定位并处理：

- 使用 UTF-8 无 BOM 重写文件
- 重新编译验证通过

这个问题说明后续所有 Java 源文件都必须统一按：

- `UTF-8 无 BOM`

保存，否则中文修复过程中还会重复触发同类问题。
